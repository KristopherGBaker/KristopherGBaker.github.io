Shikisha: Building LLM Workflows in Swift

Update (June 10, 2026): A week after this post went up, WWDC26 expanded Apple's Foundation Models framework into much of what I describe below: sessions, typed structured output, tool calling, image input, and a public LanguageModel protocol that can back a session with the on-device model, Private Cloud Compute, or third-party models.

If you are targeting current Apple platforms, I would start there first.

Shikisha still has value for older OS versions, a fuller retrieval stack, StateGraph workflows, and offline testing, but Apple's framework now covers a much larger portion of the workflow surface than it did when I originally wrote this post.

My own advice from later in the article still applies: when the system framework covers your workflow, use it.

I have been building more AI-assisted tools in Swift lately, and I kept hitting the same edge. Calling an LLM from Swift is easy. Keeping the workflow around that call understandable and maintainable is the hard part, and that part shows up the moment a feature becomes a real system: prompts, structured output, retries, streaming, tools, retrieval, memory, tests, and a UI that should still feel like a normal Apple-platform app.

That is why I started building Shikisha, a Swift-native take on LangChain-style workflows for macOS and iOS. It is still early, currently 0.1.0, but it has reached the point where I reach for it in real projects instead of just sketching examples around it.

The Problem Was the Glue

Most LLM demos start in a reasonable place: send a few messages, get text back, print the answer. That is enough for a toy chatbot. It is not enough for the tools I have been building.

JPResume, for example, turns a western resume into Japanese 履歴書 and 職務経歴書 formats. The model is useful there, but only in places. I want deterministic parsing where possible, validation between stages, structured JSON instead of a plausible-looking paragraph, and a way to inspect intermediate artifacts before they become a PDF I might send to a company. Once a tool has those constraints, the model call is just one step in a larger system.

I could always build that system in Swift, but each project carried its own pile of glue: one-off provider wrappers, handwritten JSON parsing, ad hoc retries, duplicated streaming adapters, separate mocks for tests. None of it was hard, but every new project felt like starting over. I wanted one vocabulary that works across small chains, RAG pipelines, tool-using agents, and app UI, so those pieces become composable parts instead of per-project scaffolding.

A Swift Shape for LangChain Ideas

Shikisha borrows ideas from LangChain, but it is written for Swift rather than translated line by line. Most AI tooling assumes Python, which is fine for experiments and backend services. Apple-platform apps eventually need the workflow to coexist with SwiftUI, structured concurrency, app state, testing, and sandboxing, and to live inside the app instead of behind a separate Python orchestration backend. I wanted LLM workflows to feel like normal typed application code, not a scripting layer bolted on the side.

The core idea is one protocol:

public protocol Runnable<Input, Output>: Sendable {
    associatedtype Input: Sendable
    associatedtype Output: Sendable
    func invoke(_ input: Input) async throws -> Output
}

Prompt templates, chat models, output parsers, retrievers, and whole chains can all be Runnable, so a prompt-model-parser pipeline reads top to bottom:

let chain = ChatPromptTemplate.fromTuples([
    .system("Answer concisely."),
    .human("{question}")
])
.pipe(model)
.pipe(StringOutputParser())

let answer = try await chain.invoke(["question": "What is Swift Concurrency?"])

The important part is not the operator, it is the contract. If a prompt produces messages and a model accepts messages, the compiler can check that connection. When a chain works, it can be invoked, batched, retried, wrapped with callbacks, or tested with a fake model in the same style as any smaller piece.

Under that contract Shikisha leans on the Swift tools I would expect to reach for in 2026: structured concurrency, Sendable types, AsyncSequence streaming, Codable provider payloads, and Swift Package Manager for installation.

What Is Included So Far

The current release covers the pieces I keep needing:

  • Models and prompts: chat adapters for OpenAI, Anthropic, Google Gemini, and Ollama; prompt templates; output parsers with structured-output helpers.
  • Retrieval: document loaders, text splitters, embeddings, and vector stores; RAG retrievers (vector, BM25, hybrid, MMR, and several query-rewriting strategies); incremental indexing so changed documents do not force a full re-embed.
  • Agents and workflows: conversation memory, typed tools with a tool-calling agent loop, and StateGraph for cyclic or stateful workflows.
  • Observability and testing: callbacks for tracing, usage, and cost; FakeChatModel and local embeddings for offline tests.

The list is long, but the pieces share a shape. A RAG chain and a one-question prompt are not separate worlds; they are different arrangements of the same few ideas. You can run the example suite without an API key:

swift run ShikishaExamples
swift run ShikishaExamples basicChain
swift run ShikishaExamples ragPipeline

Tests and demos should not depend on a live provider when the behavior under test is the workflow itself.

The Documentation Is Part of the Project

I recently added full DocC documentation and tutorials, which changed how the project feels. There are conceptual guides for Swift developers new to LLM patterns, a feature article for each major piece, and tutorials that build real things: a streaming chatbot with memory, a RAG app over your own documents, a tool-using agent, a small coding agent with file tools, and a SwiftUI app around that agent.

The coding-agent tutorial is a good example of why I wanted this in Swift. A surprising amount of an AI coding assistant is just a model, a few file tools, and a loop. The details still matter (sandbox paths, typed tool arguments, memory, deciding when the model can act), but once they have names the loop stays small enough to read:

let agent = ToolCallingAgent(
    model: model,
    tools: [readFileTool, editFileTool],
    maxIterations: 8
)

let result = try await agent.run([
    SystemMessage(content: "You are a coding assistant working in this project."),
    HumanMessage(content: "Summarize the TODOs in this project.")
])

Moving that from a command-line experiment into a SwiftUI app does not mean rewriting the mental model in another language.

Where I Am Already Using It

JPResume is the first public project where Shikisha fits naturally. Its pipeline needs model calls, structured output, validation, and reviewable intermediate files, and Shikisha lets me express those stages without each one owning its own provider wrapper or parser convention. It does not remove the hard product decisions, like what the model is allowed to infer or how strict validation should be. It just gives the workflow a better foundation.

I am also using it in a second project I will write about later, which leans on a different side of the library: agent behavior, tools, and app integration. That has been the more useful test. Abstractions tend to prove themselves not when the first demo works, but when they survive a second project with different pressure.

What Still Feels Early

Shikisha is a young Swift package, and the API may still move as the real use cases sharpen. There are also times the older approach is fine: if all you need is one provider call behind one button, a small direct URLSession call can be the right answer. Shikisha starts to matter when the workflow becomes more than the call: a repeatable chain, retrieval, structured output, tool execution, memory, tracing, retries, streaming, or tests that should run without hitting a provider. I am not trying to make every AI feature look elaborate, just to make the elaborate ones less fragile.

The part I care about is that the same abstractions now survive across command-line experiments, SwiftUI apps, retrieval pipelines, typed tool execution, and local tests. That is when a library starts becoming infrastructure instead of a demo.

Docs: krisbaker.com/Shikisha/documentation/shikisha Repo: github.com/KristopherGBaker/Shikisha

If you build Swift apps and have been curious about adding LLM workflows without leaving the Apple-platform toolchain, Shikisha is my current answer. It is still early, but already past being just an experiment.
